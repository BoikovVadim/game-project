# Логика турниров

Этот файл описывает фактическую логику турниров в проекте по текущему коду.

## Основные файлы

- `backend/src/tournaments/tournaments.service.ts` — главный источник бизнес-логики турниров.
- `backend/src/tournaments/tournaments.controller.ts` — HTTP API для турниров.
- `backend/src/tournaments/tournament.entity.ts` — модель турнира.
- `Frontend/src/components/Profile.tsx` — пользовательский UI турниров.
- `Frontend/src/components/Admin.tsx` — админский список всех участий в турнирах.

## Что такое турнир в системе

Сущность `Tournament` хранит:

- `id`
- `createdAt`
- `status`: `waiting` | `active` | `finished`
- `gameType`: `training` | `money`
- `leagueAmount`: ставка лиги для денежных турниров
- `playerOrder`: массив `userId`, который задает слоты игроков
- `players`: связь many-to-many с пользователями

Главное правило: пары определяются только по слотам в `playerOrder`.

- полуфинал 1: `slot 0 vs slot 1`
- полуфинал 2: `slot 2 vs slot 3`
- старые турниры с 2 игроками трактуются как частный случай той же сетки: считается только первая полуфинальная пара, а итог такого матча не даёт отдельную «логику 1v1», а показывает выигранный полуфинал и ожидание следующего этапа/соперника

Если в паре нет второго игрока, это не считается победой. Такое состояние считается ожиданием соперника.
Если второй слот занят, но у соперника ещё нет валидного `tournament_progress` для сравнения, это тоже не даёт автопроход дальше: до реального head-to-head или общего timeout пары состояние остаётся ожиданием.

## Режимы турниров

### Training

- Турнир без ставки.
- Создается или переиспользуется через `startTraining(userId)`.
- Игрок добавляется в первый доступный турнир `training` со статусом `waiting` или `active`, если там меньше 4 игроков и его там еще нет.
- Если подходящего турнира нет, создается новый.
- Для нового турнира сразу генерируются:
  - 10 вопросов полуфинала 1 (`roundIndex = 0`)
  - 10 вопросов полуфинала 2 (`roundIndex = 1`)

### Money

- Турнир со ставкой лиги.
- Создается или переиспользуется через `joinOrCreateMoneyTournament(userId, leagueAmount)`.
- Перед входом проверяются:
  - доступность лиги
  - минимальный баланс
  - победы в предыдущей лиге для открытия более высокой
- После входа:
  - деньги списываются с баланса
  - создается escrow-запись со статусом `held`
- Если списание не удалось, вход в турнир откатывается.

## Вход игрока в турнир

Для обоих режимов схема одинаковая:

1. Ищется подходящий турнир.
2. Игрок добавляется в `players`.
3. Его `userId` добавляется в конец `playerOrder`.
4. Пишется `tournament_entry`.
5. Если игрок закрыл пару в полуфинале, запускается синхронизация старта раунда для этой пары.

Слоты занимаютcя последовательно:

- игрок 1 -> slot 0
- игрок 2 -> slot 1
- игрок 3 -> slot 2
- игрок 4 -> slot 3

## Вопросы

### Полуфиналы

- На турнир генерируются 20 уникальных вопросов.
- `Полуфинал 1` получает свои 10 вопросов.
- `Полуфинал 2` получает другие 10 вопросов.
- Вопросы двух полуфиналов внутри одного турнира не должны пересекаться.

### Финал

- Финальные вопросы не создаются сразу.
- Они создаются лениво в `getTrainingState`, только когда игрок реально выиграл полуфинал.
- Финал получает отдельные 10 вопросов, которые не должны повторять уже созданные полуфинальные вопросы этого турнира.

### Тайбрейки

- Каждый тайбрейк — это еще 10 вопросов.
- Полуфинальные тайбрейки хранятся начиная с `roundIndex = 3`, `4`, `5` и так далее.
- Если счет в полуфинале равный и оба игрока уже дошли до соответствующего тайбрейка, система либо переиспользует уже созданный набор вопросов этого тайбрейка, либо генерирует новый.
- Новый тайбрейк не должен повторять вопросы, уже созданные в этом турнире для полуфиналов, финала и предыдущих тайбрейков.

## Прогресс игрока

Основная таблица состояния игрока в турнире — `tournament_progress`.

Дополнительная таблица Variant B для timeout-развязок — `tournament_round_resolution`.

Она хранит явный результат пары/раунда, когда timeout уже канонически зафиксирован:

- `tournamentId`
- `stage`: `semi` | `final`
- `pairIndex`: индекс пары (`0` или `1` для полуфиналов, `0` для финала)
- `roundNumber`: `0` для базового раунда, `1+` для тайбрейков
- `slotAUserId` / `slotBUserId`
- `outcome`: `slotA_wins` | `slotB_wins` | `both_lost`
- `reason = timeout`
- `winnerUserId` / `loserUserId`
- `sharedRoundStartedAt`
- `deadlineAt`
- `resolvedAt`
- `source`: `cron` | `backfill`

Ключевые поля:

- `questionsAnsweredCount`
- `currentQuestionIndex`
- `correctAnswersCount`
- `semiFinalCorrectCount`
- `answersChosen`
- `tiebreakerRoundsCorrect`
- `finalTiebreakerRoundsCorrect`
- `roundStartedAt`
- `leftAt`
- `lockedAnswerCount`

### Что делает `setProgress(...)`

- нормализует `answersChosen`
- не дает уменьшать прогресс
- не дает переписывать уже заблокированные ответы
- пересчитывает количество верных ответов по фактически выбранным вариантам
- фиксирует:
  - базовый счет полуфинала
  - очки тайбрейков полуфинала
  - очки тайбрейков финала
- обновляет `roundStartedAt`, если игрок пересек границу следующей фазы
- повторно читает свежие данные перед `save`, чтобы не затереть более новый прогресс из параллельного запроса
- после сохранения вызывает автозавершение `tryAutoComplete(...)`

### Anti-cheat защита

В `setProgress` реализованы несколько защит:

- нельзя переписать уже зафиксированные ответы
- не-final save не может перескочить слишком далеко вперед
- ответы с клиента склеиваются с уже заблокированными данными из базы
- финальное подтверждение (`answerFinal`) увеличивает `lockedAnswerCount`

## Как определяется дедлайн

В проекте используется дедлайн 24 часа на текущий общий раунд пары.

- `ROUND_DEADLINE_HOURS = 24`
- общий старт раунда считается от обоих игроков пары
- дедлайн рассчитывается от общего старта через `getRoundDeadline(...)`
- timeout-результат пары можно фиксировать только после подтверждённого общего дедлайна пары, а не просто по недоотвеченным вопросам
- после подтверждённого timeout canonical writer сначала пишет строку в `tournament_round_resolution`, и только потом синхронизирует `tournament_result` / `tournament.status`

Важно:

- если второй финалист пришел позже, таймер финала стартует только когда финальная пара реально сформировалась
- если игрок уже закончил текущий раунд, таймер для него не должен ломать дальнейшую логику

## Как считается полуфинал

Полуфинал считается только внутри своей пары по слотам.

Источник истины:

- игрок
- его соперник по `playerOrder`
- их `tournament_progress`
- если для этой пары уже существует строка в `tournament_round_resolution`, она имеет приоритет над повторным вычислением по `progress`

Функция `getMoneySemiResult(...)` возвращает одно из значений:

- `won`
- `lost`
- `tie`
- `incomplete`

Особенности:

- если соперника нет, возвращается `incomplete` с `noOpponent = true`
- если у соперника нет валидного `progress`, это тоже не считается победой и не переводит игрока в финал автоматически
- ничья определяется только внутри конкретной пары
- при равенстве включается цепочка тайбрейков
- если дедлайн уже истек, а один игрок успел закончить нужный тайбрейк, а второй нет, победа/поражение доопределяются автоматически
- если дедлайн уже истек и оба не закончили текущий общий раунд пары, только тогда возможен сценарий «оба проиграли по таймауту»; для тайбрейка целевой порог берётся по текущему доп. раунду, а не всегда по первым 10 вопросам

## Как считается финал

Финал возможен только когда турнир реально дошел до финальной стадии.

Функция `getFinalResult(...)` возвращает:

- `won`
- `lost`
- `tie`
- `incomplete`

Логика:

1. Определяется второй финалист из противоположной полуфинальной пары.
   - сначала читается `tournament_round_resolution` для противоположной полуфинальной пары;
   - если explicit resolution ещё нет, используется fallback-вычисление по `tournament_progress`.
2. Проверяется, что оба дошли до финала.
3. Из общего числа правильных ответов вычитаются:
   - базовый полуфинал
   - полуфинальные тайбрейки
4. Сравнивается финальный базовый счет.
5. Если он равный, сравниваются раунды финальных тайбрейков.
6. Если по финальной паре уже есть explicit timeout-resolution, он имеет приоритет над повторной догадкой по прогрессу.

## Когда турнир считается завершенным

Завершение происходит несколькими путями.

### Автозавершение после ответа

`tryAutoComplete(...)` вызывается после каждого сохранения прогресса.

Что делает:

- ждет, пока оба игрока пары ответят минимум на 10 вопросов
- определяет победителя/проигравшего полуфинала
- дальше использует ту же общую модель этапов для всех турниров:
  - сначала фиксирует проигравшего полуфинала
  - затем ждет финал
  - после определения победителя финала пишет результаты
  - переводит турнир в `finished`

### Автозавершение по таймеру

Cron раз в минуту запускает:

- `processAllExpiredEscrows()`
- `cancelUnfilledTournaments()` — сейчас пустой no-op
- `closeTimedOutRounds()`

`closeTimedOutRounds()`:

- проверяет пары в полуфиналах и финале
- если общий дедлайн пары вышел, а кто-то не закончил текущий раунд:
  - завершивший раунд побеждает
  - оба могут проиграть, если оба не закончили
  - в финале есть отдельная логика для случая `0 correct`
- при любом таком timeout-исходе сначала выполняется idempotent upsert в `tournament_round_resolution`
- если турнир окончательно разрешен, он переводится в `finished`
- для `money` сразу запускается обработка escrow

## Приоритет чтения Variant B

Ключевые read-path теперь используют один и тот же порядок:

1. Сначала читают `tournament_round_resolution`.
2. Если явной timeout-resolution для нужной пары/финала нет, только тогда fallback-но считают по `tournament_progress`.

Это касается прежде всего:

- `getMyTournaments(...)`
- `getTrainingState(...)`
- `getTournamentState(...)`
- `getTournamentBracket(...)`
- helper-ов `didUserWinSemiFinal(...)`, `computeSemiResult(...)`, `isPlayerInFinalPhase(...)`

Именно поэтому после cron-resolution список турниров, история, bracket и доступ к финалу больше не должны жить разными догадками о том, кто «как будто бы» прошёл дальше.

## Денежная логика и escrow

Escrow используется только в `money`-режиме.

После входа в турнир:

- у игрока списывается ставка
- создается `tournament_escrow` со статусом `held`

После завершения турнира:

- `processTournamentEscrow(tournamentId)` либо выплачивает приз победителю
- либо делает возврат всем участникам, если победитель не определился

Правила:

- если победитель один и есть валидный контекст финала, выплачивается выигрыш
- если победитель не определен, выполняется refund
- повторные выплаты и возвраты защищены от дублей

## Логика active / history

Главное место, где это определяется: `getMyTournaments(userId, mode, currentTournamentId)`.

Это центральная функция пользовательского списка турниров.

Она делает сразу много вещей:

1. Нормализует старые записи:
   - `gameType = null` + `leagueAmount != null` -> `money`
   - `gameType = null` + `leagueAmount = null` -> `training`
   - ошибочно помеченные `training` с ненулевой ставкой -> `money`
2. Запускает backfill связей участников.
3. Для `money` дополнительно:
   - обрабатывает просроченные escrow
   - синхронизирует участников из entry
4. Собирает ID турниров пользователя из нескольких источников:
   - `tournament_progress`
   - `tournament_entry`
   - join по `players`
   - `playerOrder`
   - raw SQL fallback для camelCase/snake_case схем
5. Загружает турниры и режет их по режиму `training` / `money`.
6. Собирает прогресс, дедлайны, результаты, даты завершения.
7. Пересчитывает `tournament_result`.
8. При необходимости:
   - переводит турниры в `finished`
   - откатывает `finished -> waiting`, если фактически там еще ничья
9. Возвращает списки:
   - `active`
   - `completed`

### Как считается текст результата

Функция `getResultLabel(...)` возвращает пользовательский статус:

- `Ожидание соперника`
- `Этап не пройден`
- `Победа`
- `Поражение`
- `Время истекло`

Она учитывает:

- число реальных игроков
- результат полуфинала
- дошел ли игрок до финала
- успел ли ответить вовремя
- есть ли соперник в паре

### Как определяется, идет турнир в историю или в активные

Функция `belongsToHistory(...)` — единая точка решения.

Ключевые правила:

- `finished` не всегда значит история
- если статус по смыслу `Ожидание соперника`, турнир может остаться в активных
- если игрок выиграл полуфинал, но еще не дошел до финала, турнир:
  - остается в `active`
  - при этом отдельной записью может появиться в истории как пройденный полуфинал
- при истекшем времени турнир уходит в историю

### Специальный сценарий "выиграл полуфинал, финал не начат"

В проекте есть отдельная поддержка этого состояния:

- турнир остается доступным в активных
- в истории может появиться запись о завершенном полуфинале с результатом `Победа`

## Как формируется элемент списка турниров

`toItem(...)` собирает DTO для фронта.

В элемент попадают:

- базовые поля турнира
- `deadline`
- `userStatus`
- `stage`
- `resultLabel`
- `roundForQuestions`
- `questionsAnswered`
- `questionsTotal`
- `correctAnswersInRound`
- `completedAt`
- `roundStartedAt`
- вложенный объект `tournament`:
  - `id`
  - `name`
  - `type`
  - `status`
  - `leagueAmount`

## Состояние турнира для продолжения игры

### `getTournamentState(...)`

Нужен для кнопки "Продолжить игру".

Возвращает:

- слот игрока
- номер полуфинала
- позицию в паре
- `deadline`
- активный раунд тайбрейка
- вопросы тайбрейка, если они уже должны быть доступны

Особенность:

- если турнир формально `finished`, но игрок выиграл полуфинал и еще не прошел финал, доступ к продолжению сохраняется

### `getTrainingState(...)`

Возвращает полный игровой state:

- полуфинальные вопросы
- финальные вопросы
- тайбрейк-вопросы
- прогресс
- ответы игрока
- ответы соперника по раундам
- `semiResult`
- текущий дедлайн

Там же:

- лениво создаются финальные вопросы
- запускается таймер у финалистов, когда оба действительно вошли в фазу финала
- собираются данные для модалки просмотра вопросов

## Фронтенд логика

Основной экран игрока: `Frontend/src/components/Profile.tsx`.

### Основные действия

- `startTraining()` — старт тренировки
- `joinTournament(leagueAmount)` — вход в денежный турнир
- `continueTournament(tournamentId)` — продолжение текущего турнира
- `openBracket(tournamentId, source)` — модалка сетки
- `openQuestionsReview(tournamentId, round)` — модалка вопросов

### Что показывает игроку фронтенд

Во вкладке игр фронтенд получает от `/tournaments/my`:

- `active`
- `completed`

И далее показывает:

- `ID турнира`
- этап
- дедлайн
- результат
- вопросы в формате `answered / total / correct`
- модалку сетки турнира
- модалку вопросов турнира

## Админка

Админка не имеет отдельной турнирной логики расчета.

Она использует уже готовую пользовательскую логику:

- `getAllParticipationsForAdmin()` берет список пользователей
- для каждого пользователя вызывает:
  - `getMyTournaments(userId, 'training')`
  - `getMyTournaments(userId, 'money')`

То есть админская таблица показывает ту же турнирную картину, что и пользователь, но по всем игрокам сразу.

## Служебные backfill-процессы

В проекте много защит для старых и поврежденных данных.

Основные:

- `backfillTournamentPlayersFromEntry()`:
  - восстанавливает `tournament_players_user` из `tournament_entry`
  - восстанавливает `tournament_players_user` из `tournament_progress`
  - восстанавливает `tournament_players_user` из `playerOrder`
  - восстанавливает `tournament_entry` из `playerOrder`
- backfill `gameType` для старых турниров
- backfill `semiFinalCorrectCount`
- backfill рассинхрона `9/10`, `19/20`
- backfill дат завершения результатов

## Ключевые выводы

Если нужно менять турнирную логику, почти всегда смотреть в таком порядке:

1. `tournaments.service.ts`
2. `getMyTournaments(...)`
3. `getMoneySemiResult(...)`
4. `getFinalResult(...)`
5. `getResultLabel(...)`
6. `belongsToHistory(...)`
7. `getTrainingState(...)`
8. `setProgress(...)`
9. `tryAutoComplete(...)`
10. `closeTimedOutRounds(...)`

Если нужно понять, почему турнир отображается не там или не с тем статусом, источник истины почти всегда один: `getMyTournaments(...)`.
